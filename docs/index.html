Eliminating Multipliers with Log / LNS Arithmetic

A general-purpose look at trading hardware multipliers for a small log-domain ROM. Two RTL implementations of the same multiply-compare kernel (A·B + C·D) > Vth (10-bit inputs) — one with real multipliers, one multiplier-free via a K=2 logarithmic (LNS) datapath — synthesized and compared on the sky130 HD standard-cell library.

Dropping the two multipliers for the K=2 log datapath cuts area 57% and estimated power 58%, at a 2.84% accuracy cost vs. exact math.

RTL structure — modules & signals

Datapath — target use case (A·B − C·D) > Vth

💡 What exactly is the “LNS add” ROM? It’s a 1-input table F(d) = log₂(1 + 2−d) indexed only by d = |x − y|, and the result stays in the log domain — it is not 2x − 2y. See the derivation ↓

Step-by-step — the LNS add for row 1 (C·D = 12·40)

Exactly what the multiplier-free path does for worked-example row 1 (A=25, B=30, C=12, D=40): leading-one-detector log conversion (lod5.v), log-add = multiply, then the LNS add (lns_add.v) with the F ROM (lns_ftable.v). Logs are in quarter-log₂ units (integer 4·log₂).

  A=25  B=30  C=12  D=40   (Vth=600)        logs in 1/4-log2 units

  1) log-convert   L(v) = 4*floor(log2 v) + (top 2 mantissa bits)
       L(25) = 4*4 + 2 = 18       25 = 11001b
       L(30) = 4*4 + 3 = 19       30 = 11110b
       L(12) = 4*3 + 2 = 14       12 = 1100b
       L(40) = 4*5 + 1 = 21       40 = 101000b

  2) multiply = add the logs   (adding logs multiplies the operands)
       X = L(25)+L(30) = 37      ->  A*B ~ 2^(9.25) = 609     (true 750)
       Y = L(12)+L(40) = 35      ->  C*D ~ 2^(8.75) = 431     (true 480)

  3) LNS add   (combine the two products, still in the log domain)
       d = |X - Y| = |37 - 35| = 2
       F(d) = ROM[2] = 3                         <-- the one table lookup
       s = max(X,Y) + F(d) = 37 + 3 = 40

  4) result    S_hat = 2^(10) = 1024              (true A*B+C*D = 1230,  err -16.7%)

  5) compare   Lv = L(600) = 36  ->  V_hat = 512
       s=40 > Lv=36   ->  out = 1       (exact: 1230 > 600 -> 1)   match

How the “LNS add” ROM works

In a logarithmic number system every value is carried as its base-2 log, so multiply is free — you add the logs: x = log₂A + log₂B = log₂(A·B). The hard part is adding two values, and that is the one operation the ROM exists for. Here is the exact identity it implements (base 2, because the leading-one detector yields log₂ directly):

  A·B = 2^x           C·D = 2^y            x = log2 A + log2 B

  2^x + 2^y                       (assume x ≥ y)
    = 2^x · ( 1 + 2^(y−x) )
    = 2^max(x,y) · ( 1 + 2^−d )         d = |x − y|

  s = log2( 2^x + 2^y )
    = max(x,y) + log2( 1 + 2^−d )
    = max(x,y) + F(d)                   F(d) = log2(1 + 2^−d)   ← the ROM

The key move is factoring out the larger term: 2x+2y = 2max(x,y)·(1+2−d). What’s left, F(d) = log₂(1 + 2−d), depends on the single variable d = |x − y| — so the ROM is a small 1-D table, not a 2-D (x, y) surface, and the sum never leaves the log domain (s is compared against log₂(Vth) directly; the linear value 2x − 2y is never formed).

Logs are carried in quarter-log₂ units (the integer 4·log₂), so the ROM is indexed by the integer |X−Y| and outputs round(4·F) — 0…4. The whole table is [4, 4, 3, 3, 2, 2, 2, 2, 1, …, 0] (79 entries, d = 0…78), i.e. a 3-bit value. That is the entire “F(d) ROM” drawn in the datapath above.

Subtraction is the hard cousin. A true LNS subtract would use F₋(d) = log₂(1 − 2−d) (plus a sign bit) — still a 1-input table of |x − y|, but it diverges to −∞ as d → 0 (catastrophic cancellation when A·B ≈ C·D), exactly where a small K is weakest. The threshold compare (A·B − C·D) > Vth sidesteps it by rearranging to an add, A·B > C·D + Vth, reusing this same F(d) = log₂(1 + 2−d) ROM.

Inside the LUT — the correction Vth′(C,D)

Take the sub-block that folds the threshold into a product: input log₂(C·D), output log₂(C·D + Vth). Factor out the input:

  log₂(C·D + Vth)  =  log₂(C·D)  +  Vth′(C,D)

     1st term :  log₂(C·D)                                     ← a wire (the input)
     2nd term :  Vth′(C,D) = log₂( 1 + 2^( log₂Vth − log₂(C·D) ) )   ← the LUT

  since   log₂(C·D) + log₂(1 + Vth/(C·D))  =  log₂(C·D + Vth)

The 1st term is a wire (the input, carried into an adder); the 2nd term is the LUT — name it Vth′(C,D) = log₂(1 + 2log₂Vth − log₂(C·D)), the log-domain amount by which Vth lifts log₂(C·D) (quarter-log₂ units). So the whole block is just out = log₂(C·D) + Vth′(C,D). Vth stays programmable: it is a runtime input, and the LUT below is Vth-independent (the general correction addressed by the log-gap), so changing Vth just slides where each C·D lands on it — it is not baked into the table.

On an ASIC there is no LUT primitive — synthesis maps this correction to 6 standard cells (33.8 µm², <1% of the log detector). The table is the design's general, bounded correction addressed by the log-gap d = |log₂(C·D) − log₂(Vth)|, value Vth′ (0…4, a 3-bit output). In the full A·B+C·D detector the same table combines the two products instead, and when C·D < Vth the roles swap so the address stays ≥ 0.

Transfer function  Vth′ vs log₂(Vth), for two product levels

Sweeping the programmable Vth. x is the runtime threshold log₂(Vth); the two quantized staircases (quarter-log₂ steps — the actual LUT function, not a smooth curve) are two fixed product levels C·D, 4× apart. Each rises with Vth: the knee (dot) is at Vth = C·D (Vth′ = log₂2 = 1). For Vth ≪ C·D the correction ≈ 0 (the threshold is negligible next to the product); for Vth ≫ C·D it rises toward log₂(Vth) − log₂(C·D) (the threshold dominates). A 4× larger C·D shifts the knee right by log₂4 = 2 — a bigger product needs a proportionally bigger Vth before the threshold starts to matter.

Behavioral RTL  rtl/lns_ftable.v

module lns_ftable(input [7:0] d, output reg [3:0] f);   // generated by model.py
  always @* case (d)
    8'd0  : f = 4'd4;
    8'd1  : f = 4'd4;
    8'd2  : f = 4'd3;
    8'd3  : f = 4'd3;
    ...                     // 79 entries (d = 0 … 78)
    8'd13 : f = 4'd1;
    default : f = 4'd0;
  endcase
endmodule

Gate-level netlist  after yosys + ABC (sky130)

module lns_ftable(input [7:0] d, output [3:0] f);   // 6 cells · 33.8 µm²
  wire _0_, _1_, _2_;
  sky130_fd_sc_hd__or4_1    g0 (.A(d[7]), .B(d[4]), .C(d[5]), .D(d[6]), .X(_0_));
  sky130_fd_sc_hd__nor2_1   g1 (.A(d[3]), .B(d[1]), .Y(_1_));
  sky130_fd_sc_hd__nor2_1   g2 (.A(d[2]), .B(d[1]), .Y(_2_));
  sky130_fd_sc_hd__a211oi_1 g3 (.A1(d[2]), .A2(d[1]), .B1(_0_), .C1(_1_), .Y(f[0]));
  sky130_fd_sc_hd__nor3_1   g4 (.A(d[3]), .B(_0_), .C(_2_), .Y(f[1]));
  sky130_fd_sc_hd__nor4_1   g5 (.A(d[3]), .B(d[2]), .C(d[1]), .D(_0_), .Y(f[2]));
  assign f[3] = 1'b0;                                // F ≤ 4, so bit 3 is always 0
endmodule

Logic-gate schematic

Two logic levels: or4_1/nor2_1 build the internal terms (_0_ = OR of the high address bits, i.e. “d is large”; _1_, _2_), then a211oi_1/nor3_1/nor4_1 produce the three output bits f[0..2]; f[3] is tied low.
