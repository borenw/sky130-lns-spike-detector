Eliminating Multipliers with K=1 Log / LNS Arithmetic

A general-purpose look at trading hardware multipliers for a small log-domain ROM. Two RTL implementations of the same multiply-compare kernel (A·B + C·D) > Vth (12-bit inputs) — one with real multipliers, one multiplier-free via a K=1 logarithmic (LNS) datapath — synthesized and compared on the sky130 HD standard-cell library.

Dropping the two multipliers for the K=1 log datapath cuts area 70% and estimated power 74%, at a 5.60% accuracy cost vs. exact math.

RTL structure — modules & signals

Datapath — target use case (A·B − C·D) > Vth

💡 What exactly is the “LNS add” ROM? It’s a 1-input table F(d) = log₂(1 + 2−d) indexed only by d = |x − y|, and the result stays in the log domain — it is not 2x − 2y. See the derivation ↓

Step-by-step — the LNS add for row 1 (C·D = 12·40)

Exactly what the multiplier-free path does for worked-example row 1 (A=25, B=30, C=12, D=40): leading-one-detector log conversion (lod5.v), log-add = multiply, then the LNS add (lns_add.v) with the F ROM (lns_ftable.v). Logs are in half-log₂ units (integer 2·log₂).

  A=25  B=30  C=12  D=40   (Vth=600)        logs in half-log2 units

  1) log-convert   L(v) = 2*floor(log2 v) + (next bit below the leading 1)
       L(25) = 2*4 + 1 = 9        25 = 11001b
       L(30) = 2*4 + 1 = 9        30 = 11110b
       L(12) = 2*3 + 1 = 7        12 = 1100b
       L(40) = 2*5 + 0 = 10       40 = 101000b

  2) multiply = add the logs   (adding logs multiplies the operands)
       X = L(25)+L(30) = 18      ->  A*B ~ 2^(9) = 512     (true 750)
       Y = L(12)+L(40) = 17      ->  C*D ~ 2^(8.5) = 362     (true 480)

  3) LNS add   (combine the two products, still in the log domain)
       d = |X - Y| = |18 - 17| = 1
       F(d) = ROM[1] = 2                         <-- the one table lookup
       s = max(X,Y) + F(d) = 18 + 2 = 20

  4) result    S_hat = 2^(10) = 1024              (true A*B+C*D = 1230,  err -16.7%)

  5) compare   Lv = L(600) = 18  ->  V_hat = 512
       s=20 > Lv=18   ->  out = 1       (exact: 1230 > 600 -> 1)   match

How the “LNS add” ROM works

In a logarithmic number system every value is carried as its base-2 log, so multiply is free — you add the logs: x = log₂A + log₂B = log₂(A·B). The hard part is adding two values, and that is the one operation the ROM exists for. Here is the exact identity it implements (base 2, because the leading-one detector yields log₂ directly):

  A·B = 2^x           C·D = 2^y            x = log2 A + log2 B

  2^x + 2^y                       (assume x ≥ y)
    = 2^x · ( 1 + 2^(y−x) )
    = 2^max(x,y) · ( 1 + 2^−d )         d = |x − y|

  s = log2( 2^x + 2^y )
    = max(x,y) + log2( 1 + 2^−d )
    = max(x,y) + F(d)                   F(d) = log2(1 + 2^−d)   ← the ROM

The key move is factoring out the larger term: 2x+2y = 2max(x,y)·(1+2−d). What’s left, F(d) = log₂(1 + 2−d), depends on the single variable d = |x − y| — so the ROM is a small 1-D table, not a 2-D (x, y) surface, and the sum never leaves the log domain (s is compared against log₂(Vth) directly; the linear value 2x − 2y is never formed).

Logs are carried in half-log₂ units (the integer 2·log₂), so the ROM is indexed by the integer |X−Y| and outputs round(2·F) — just 0, 1, or 2. The whole table is [2, 2, 1, 1, 1, 0, 0, 0, 0, …, 0] (47 entries, d = 0…46), i.e. ~2 bits out. That is the entire “F(d) ROM” drawn in the datapath above.

Subtraction is the hard cousin. A true LNS subtract would use F₋(d) = log₂(1 − 2−d) (plus a sign bit) — still a 1-input table of |x − y|, but it diverges to −∞ as d → 0 (catastrophic cancellation when A·B ≈ C·D), exactly where K=1 is weakest. The threshold compare (A·B − C·D) > Vth sidesteps it by rearranging to an add, A·B > C·D + Vth, reusing this same F(d) = log₂(1 + 2−d) ROM.
